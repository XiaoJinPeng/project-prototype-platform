# 国际化地址模型与地址组件规则

## 1. 文档定位

本文用于统一 RIMO 平台在多国家、多租户场景下的地址字段模型、地址组件模式、前后端处理口径和页面适用范围。

本文解决的问题是：

1. 客服系统为租户配置所属国家或地区后，营运端地址表单如何跟随切换。
2. 不同国家地址结构不一致时，营运端如何避免每个页面各写一套地址组件。
3. 如何在不做重型国际地址引擎的前提下，满足当前 SaaS 多地区租户的使用需求。

本文为平台级规则文档。后续所有涉及地址录入、地址展示、地址搜索、地址快照、地图定位的需求，均按本文执行。

---

## 2. 背景与问题

当前平台是 SaaS 多租户架构。客服系统在创建租户时，会确定该租户所属的国家或地区。营运端登录后，门市、据点、会员、企业等页面都会涉及地址录入。

现有原型中，部分页面已经采用台湾行政区级联选择方式录入地址。该方式只适用于台湾口径，不适用于香港、新加坡、马来西亚、欧洲等其他国家或地区。

如果继续沿用单一台湾级联地址组件，会出现以下问题：

1. 不同国家行政层级不同，无法强行套用同一套“省市区”逻辑。
2. 不同页面可能各自定义一套地址字段，后端难以统一存储。
3. 地址字段如果只存一个完整字符串，后续搜索、地图、导出、快照、统计和页面展示会越来越混乱。
4. “行政地址地区”和“业务运营地区”容易混用，导致甲租乙还地区、行销适用地区与真实通讯地址混在一起。

因此，本期统一采用**两种地址模式**，并统一字段模型。

---

## 3. 本期决策结论

### 3.1 地址模式只保留两种

本期平台地址模式固定为两种：

1. **台湾模式（TW）**
2. **国际通用模式（GLOBAL）**

不做国家模板中心，不做多国家动态地址引擎，不维护每个国家一套独立行政区规则。

### 3.2 地址模式由租户所属国家自动决定

客服系统只维护租户所属国家或地区，不额外维护独立的地址模式开关。

系统判断规则固定为：

1. 租户所属国家代码为 `TW` 时，营运端地址模式为 `TW`
2. 其他国家或地区，营运端地址模式统一为 `GLOBAL`

### 3.3 一个租户只能使用一套地址模式

同一个租户在同一时间内，只能使用一套地址模式。

营运端中所有地址相关页面，必须统一跟随该租户的地址模式，不允许：

1. 某些页面使用台湾级联
2. 某些页面又使用国际文本输入
3. 某些页面再单独出现国家下拉框让用户自己选

### 3.4 营运端大多数页面不再让用户选择国家

营运端登录后，国家或地区已经由租户确定。

因此，营运端中的地址表单默认不再提供“国家/地区”可编辑选择器。页面只按租户当前地址模式渲染地址字段。

若页面需要展示国家信息，可只读展示当前租户国家名称，不允许在表单中修改。

### 3.5 “业务地区”与“行政地址地区”必须拆开

以下内容不属于地址组件范围，不得复用地址级联或地址字段：

1. 甲租乙还地区调度费中的“地区”
2. 行销方案适用地区中的“地区”
3. 业务统计中的“地区”
4. 运营分区、站群、区域负责人中的“地区”

这些属于**业务地区模型**，不是行政地址模型。

## 4. 适用范围

本文适用于以下对象的地址录入和展示：

### 4.1 营运地址

1. 车行地址
2. 门市地址
3. 据点地址

### 4.2 通讯地址

1. 个人会员户籍地址
2. 个人会员通讯地址
3. 企业客户通讯地址
4. 后续可能新增的发票寄送地址、账单寄送地址、联系人地址

### 4.3 地址展示与搜索

1. 列表页地址展示
2. 详情页地址展示
3. 搜索条件中的地址关键字
4. 导出文件中的地址字段
5. 订单快照中的门市/据点地址文本

本文不适用于：

1. 业务地区配置
2. 甲租乙还地区费用矩阵
3. 行销方案适用地区
4. 报表区域维度
5. 地图服务商、导航服务商、地理编码服务商接入规则

## 5. 客服系统配置规则

### 5.1 客服系统必须维护租户所属国家或地区

客服系统在创建租户时，必须维护以下字段：

| 字段 | 说明 |
| --- | --- |
| `tenantCountryCode` | 租户所属国家或地区代码，例如 `TW`、`SG`、`HK`、`MY` |
| `tenantCountryName` | 租户所属国家或地区名称 |

这两个字段属于租户平台基础配置，不属于营运端用户可编辑资料。

### 5.2 地址模式判断规则

后端统一按以下逻辑计算租户地址模式：

| 条件 | 地址模式 |
| --- | --- |
| `tenantCountryCode = TW` | `TW` |
| 其他值 | `GLOBAL` |

后端在租户登录上下文、租户基础信息接口或初始化配置接口中，必须返回以下字段：

| 字段 | 说明 |
| --- | --- |
| `tenantCountryCode` | 当前租户国家代码 |
| `tenantCountryName` | 当前租户国家名称 |
| `tenantAddressMode` | 当前租户地址模式，只允许为 `TW` 或 `GLOBAL` |

前端不自行根据国家代码猜测模式，统一读取后端返回的 `tenantAddressMode`。

### 5.3 租户国家修改限制

租户所属国家或地区一旦确定，会直接影响地址组件、字段校验和主数据结构。

因此，客服系统对租户国家的修改规则如下：

1. **租户尚未产生业务数据前**，允许修改所属国家或地区。
2. **租户已产生业务数据后**，不允许直接修改所属国家或地区。

“已产生业务数据”至少包括以下任一情况：

1. 已建立车行
2. 已建立门市
3. 已建立据点
4. 已建立会员
5. 已建立企业客户
6. 已产生订单

如确需切换国家或地区，按以下方式处理：

1. 新建新租户
2. 或单独规划数据迁移项目

本期不支持对既有租户直接切换国家后自动转换历史地址结构。

### 5.4 历史租户未配置国家的兜底规则

若历史租户尚未补齐国家配置，后端临时按以下规则兜底：

1. `tenantCountryCode` 为空时，`tenantAddressMode` 默认返回 `GLOBAL`
2. 同时在管理端提示“租户国家配置缺失”

该兜底只用于避免页面无法渲染，不代表该租户已完成地址规则配置。客服系统仍需补齐租户国家资料。

## 6. 统一地址数据模型

### 6.1 所有地址对象统一字段结构

本期所有地址对象统一采用以下字段模型：

| 字段 | 说明 |
| --- | --- |
| `countryCode` | 国家或地区代码，来自租户配置 |
| `countryName` | 国家或地区名称，来自租户配置 |
| `region1Code` | 一级区域代码；有标准字典时保存，无字典时可为空 |
| `region1Name` | 一级区域名称 |
| `region2Code` | 二级区域代码；有标准字典时保存，无字典时可为空 |
| `region2Name` | 二级区域名称 |
| `region3Code` | 三级区域代码；有标准字典时保存，无字典时可为空 |
| `region3Name` | 三级区域名称 |
| `postalCode` | 邮编 |
| `addressDetail` | 详细地址 |
| `fullAddress` | 展示用完整地址，由系统按统一规则拼接 |
| `lat` | 纬度；仅营运地址强制要求 |
| `lng` | 经度；仅营运地址强制要求 |

### 6.2 字段含义必须按“通用层级”理解

`region1/2/3` 是平台统一的通用区域字段，不按某一个国家固定解释。

不同地址模式下，字段意义如下：

| 模式 | `region1` | `region2` | `region3` |
| --- | --- | --- | --- |
| `TW` | 台湾行政区第 1 层 | 台湾行政区第 2 层 | 台湾行政区第 3 层 |
| `GLOBAL` | 州 / 省 / 地区 | 城市 | 区域 / 行政区 |

### 6.3 `fullAddress` 不是手工录入字段

`fullAddress` 由系统统一拼接生成，不提供给用户手工输入。

拼接规则如下：

1. 按 `countryName + region1Name + region2Name + region3Name + postalCode + addressDetail` 顺序拼接
2. 空字段自动跳过
3. 展示时使用统一格式，不允许各页面自己拼一版地址文本

### 6.4 只存完整地址字符串不符合平台要求

后端不得只保存一个 `address` 字符串，然后丢弃结构化字段。

所有新建和编辑后的地址记录，必须同时保存：

1. 结构化字段
2. 完整展示地址

否则后续无法支撑：

1. 搜索
2. 导出
3. 地图
4. 电子围栏
5. 快照
6. 历史审计

## 7. 地址组件分类

本期前端统一维护两类地址组件，不允许每个页面各写一套地址字段组合：

### 7.1 营运地址组件

适用对象：

1. 车行
2. 门市
3. 据点

特点：

1. 使用租户地址模式渲染
2. 保存结构化地址字段
3. 支持完整地址展示
4. 支持地图定位
5. 需要保存经纬度

### 7.2 通讯地址组件

适用对象：

1. 个人会员户籍地址
2. 个人会员通讯地址
3. 企业客户通讯地址
4. 其他纯通讯场景地址

特点：

1. 使用租户地址模式渲染
2. 保存结构化地址字段
3. 不强制保存经纬度
4. 不涉及电子围栏

---

## 8. 台湾模式（TW）规则

### 8.1 适用条件

当 `tenantAddressMode = TW` 时，所有地址表单统一使用台湾模式。

### 8.2 录入组件

台湾模式地址组件统一包含以下内容：

| 字段 | 录入方式 | 必填 |
| --- | --- | --- |
| 行政区选择 | 级联选择器 | 是 |
| 邮编 | 自动带出或手工补录 | 否 |
| 详细地址 | 文本输入 | 是 |

营运地址额外包含：

| 字段 | 录入方式 | 必填 |
| --- | --- | --- |
| 经纬度 | 地图选点或手工录入 | 门市可选，据点必填 |

### 8.3 数据写入规则

台湾模式保存时：

1. 级联选择器的三个层级分别写入 `region1Name`、`region2Name`、`region3Name`
2. 如台湾行政区字典存在代码，则同步写入 `region1Code`、`region2Code`、`region3Code`
3. `countryCode` 固定写入当前租户国家代码
4. `countryName` 固定写入当前租户国家名称
5. `fullAddress` 由系统自动拼接

### 8.4 现有台湾原型兼容规则

当前服务站点管理页面已经采用台湾行政区级联方式。

本期保留该模式，不改成国际通用文本输入。

后续台湾租户新增的车行、门市、据点、会员、企业通讯地址，也统一按该模式录入。

---

## 9. 国际通用模式（GLOBAL）规则

### 9.1 适用条件

当 `tenantAddressMode = GLOBAL` 时，所有地址表单统一使用国际通用模式。

### 9.2 录入组件

国际通用模式统一使用简化结构，不维护各国行政区字典，不做级联选择器。

组件统一包含以下字段：

| 字段 | 录入方式 | 必填 |
| --- | --- | --- |
| 州 / 省 / 地区 | 文本输入 | 是 |
| 城市 | 文本输入 | 是 |
| 区域 / 行政区 | 文本输入 | 否 |
| 邮编 | 文本输入 | 否 |
| 详细地址 | 文本输入 | 是 |

营运地址额外包含：

| 字段 | 录入方式 | 必填 |
| --- | --- | --- |
| 经纬度 | 地图选点或手工录入 | 门市可选，据点必填 |

### 9.3 数据写入规则

国际通用模式保存时：

1. “州 / 省 / 地区”写入 `region1Name`
2. “城市”写入 `region2Name`
3. “区域 / 行政区”写入 `region3Name`
4. 由于本期不维护国际行政区代码字典，`region1Code`、`region2Code`、`region3Code` 可为空
5. `countryCode` 固定写入当前租户国家代码
6. `countryName` 固定写入当前租户国家名称
7. `fullAddress` 由系统自动拼接

### 9.4 本期不做国家差异化细分

本期不再继续区分：

1. 香港模式
2. 新加坡模式
3. 马来西亚模式
4. 欧盟模式

除台湾外，全部统一走 `GLOBAL`。

## 10. 不同对象的字段要求

### 10.1 营运地址对象

| 对象 | 地址模式 | 经纬度 | 说明 |
| --- | --- | --- | --- |
| 车行 | 跟随租户地址模式 | 不强制 | 用于通讯和资料展示 |
| 门市 | 跟随租户地址模式 | 可选 | 用于门店展示、导航、资料维护 |
| 据点 | 跟随租户地址模式 | 必填 | 用于地图定位、电子围栏、自助取还车、站点判断 |

### 10.2 通讯地址对象

| 对象 | 地址模式 | 经纬度 | 说明 |
| --- | --- | --- | --- |
| 个人会员户籍地址 | 跟随租户地址模式 | 不保存 | 用于会员资料 |
| 个人会员通讯地址 | 跟随租户地址模式 | 不保存 | 用于会员联络 |
| 企业客户通讯地址 | 跟随租户地址模式 | 不保存 | 用于企业资料 |

### 10.3 订单与快照中的地址

订单不是地址主数据维护页面，但会展示和保存门市、据点地址文本。

订单、取车、还车、快照中的地址规则如下：

1. 保存当时的地址文本快照
2. 不通过当前门市 ID、据点 ID 回查最新地址来覆盖历史订单
3. 门市或据点后续改名、改地址，不反向改写历史订单快照

## 12. 展示、搜索、导出规则

### 12.1 页面展示

页面展示地址时，统一优先显示 `fullAddress`，不要求页面逐个拼接 `region1 + region2 + region3 + detail`。

如果页面空间不足，可以只展示：

1. 主名称 + `fullAddress`
2. 或主名称 + 地址截断文本

但来源仍然是统一生成的 `fullAddress`。

### 12.2 搜索

涉及地址搜索的页面，搜索口径统一按以下字段进行模糊匹配：

1. `fullAddress`
2. `addressDetail`
3. `region1Name`
4. `region2Name`
5. `region3Name`

页面不需要分别暴露所有区域字段作为搜索项，除非该页面本身有明确的区域筛选业务需求。

### 12.3 导出

导出文件中的地址字段统一至少包含：

1. 国家或地区
2. 完整地址

如需要更细粒度导出，再增加：

1. 一级区域
2. 二级区域
3. 三级区域
4. 邮编
5. 详细地址

## 13. 前后端处理要求

### 13.1 后端要求

后端必须做到以下事项：

1. 租户登录上下文中返回 `tenantCountryCode`、`tenantCountryName`、`tenantAddressMode`
2. 所有地址保存接口统一接收结构化字段，不只接收一个完整地址字符串
3. 所有地址保存接口统一生成并回写 `fullAddress`
4. 据点地址保存时校验经纬度必填
5. 历史订单、取车、还车、快照中的地址文本不得被当前主数据覆盖
6. 客服系统修改租户国家时，按本文限制校验是否允许修改

### 13.2 前端要求

前端必须做到以下事项：

1. 所有地址页面统一读取 `tenantAddressMode`
2. 不允许某些页面自行硬编码台湾级联地址，另一些页面自行写国际文本框
3. 不允许在营运端表单中再次提供国家选择器
4. `fullAddress` 不允许由页面各自随意拼接，必须走统一函数
5. 营运地址与通讯地址使用统一的两类地址组件，不再分散实现

### 13.3 接口容错要求

若接口返回的历史数据只有 `fullAddress`，结构化字段为空，则页面查看时：

1. 继续展示 `fullAddress`
2. 不阻断列表和详情展示

本期不做历史地址字符串的自动拆解回填。

## 14. 存量数据与兼容规则

### 14.1 本期不做自动拆解历史地址

已有历史资料若只存了完整地址字符串，本期不做：

1. 自动拆解成结构化字段
2. 自动反推行政区
3. 自动补齐经纬度

### 14.2 对历史数据的处理方式

历史数据处理规则如下：

1. 查看页面继续按原有 `fullAddress` 展示
2. 新建资料按新规则保存
3. 编辑并重新保存后的资料，按新规则写入结构化字段

### 14.3 本期不处理跨模式数据转换

本期不支持：

1. 台湾模式历史地址自动转国际模式
2. 国际模式历史地址自动转台湾模式
3. 直接切换租户国家后批量重算地址结构

因此，客服系统才需要对租户国家修改做前置限制。

## 15. 业务地区与地址地区的拆分规则

为避免后续开发混淆，以下规则固定：

### 15.1 地址地区

地址地区用于：

1. 通讯地址
2. 门市地址
3. 据点地址
4. 地图定位
5. 快照展示

地址地区使用本文的 `region1/2/3` 模型。

### 15.2 业务地区

业务地区用于：

1. 甲租乙还地区调度费
2. 行销方案适用地区
3. 区域统计
4. 运营分区

业务地区使用独立业务模型，不复用本文地址字段。

### 15.3 禁止事项

以下做法禁止：

1. 用门市行政区直接替代甲租乙还地区表
2. 用会员通讯地址城市去判断业务地区权限
3. 用地址级联组件去配置行销方案业务适用地区

---

## 16. 本期不做

本期明确不做以下内容：

1. 多国家独立地址模板中心
2. 各国行政区字典维护平台
3. 国际地址自动标准化服务
4. 历史地址自动拆解迁移
5. 地址智能纠错
6. 根据国家动态切换表单标签文案库
7. 同一租户内多国家地址并行录入

---

## 17. 开发落地检查清单

后续开发、联调、验收时，至少检查以下事项：

1. 客服系统新增租户时是否必填国家或地区
2. 营运端登录后是否能拿到 `tenantAddressMode`
3. 台湾租户是否统一显示台湾级联地址组件
4. 非台湾租户是否统一显示国际通用地址组件
5. 营运端表单中是否已去掉国家可编辑选择
6. 服务站点管理中的据点地址是否校验经纬度
7. 会员和企业通讯地址是否不再强制经纬度
8. `fullAddress` 是否由统一逻辑生成
9. 历史订单地址是否读取快照文本而不是当前主数据
10. 业务地区配置是否没有误用地址组件
