## 1. 算法目标

智能调度派车算法用于在订单取车前为订单匹配最合适车辆。本期算法遵循以下原则：

1. **大石块优先填空**：同车组内优先处理调度占用时长更长的订单，避免短订单先切碎车辆时间轴。
2. **只在可交付车辆中选择**：车辆必须先满足车组、门店、状态、时间窗和整备缓冲等硬约束。
3. **评分低者优先派遣**：车辆评分越低，代表车辆越需要被派出，系统优先分配低评分车辆。
4. **应用后动态更新**：同一轮优化中，每派出一辆车，系统立即在内存中模拟更新该车的预计营收、预计里程和时间占用，避免同一车辆被连续过度分配。
5. **AI 分析不替代硬规则**：AI 调度分析可以比较基线方案、提示风险并推荐动作，但不得跳过车辆池过滤、时间窗判断和应用前二次校验。

## 2. 车辆评分公式

车辆评分按照以下公式计算：

```text
车辆评分 = ((B / C) × H) + ((D / E) × I) + (1 - ((G - F) / 365) × J)
```

| 参数 | 含义 | 数据来源 |
| :--- | :--- | :--- |
| B | 已行驶里程 | 车辆档案或车辆里程记录 |
| C | 预计退役里程 | 车辆档案中按车辆单独维护 |
| D | 当月预计营收 | 服务日期在当月的预约中订单金额 + 当月已完成订单金额 |
| E | 车辆成本（每月） | 车辆档案中按车辆单独维护 |
| F | 出厂年月 | 车辆档案 |
| G | 当前年月 | 系统当前年月 |
| H | 里程权重 | 调度规则配置 |
| I | 营收权重 | 调度规则配置 |
| J | 车龄权重 | 调度规则配置 |

权重按百分比参与计算，100% 记为 1，0% 记为 0。当前默认权重为：里程权重 0%、营收权重 100%、车龄权重 0%。后台允许调整三项权重，三项合计必须等于 100%。

该公式属于按车辆自身配置基准做比例化处理：

1. `B / C` 将已行驶里程转换为“当前里程占预计退役里程的比例”。
2. `D / E` 将当月预计营收转换为“当前营收占车辆月成本的比例”。
3. `(G - F) / 365` 将出厂年月到当前年月的时间差转换为车辆年龄年数。

该口径与直接使用原始营收、原始里程、原始车龄相加不同；它先把不同量纲转换为比例后再加权。该口径不是按全车队最大值、最小值映射到 0-100 的归一化，比例值可以超过 1。

在当前默认权重下，里程权重和车龄权重均为 0%，车辆评分等价于 `1 + 当月预计营收 / 车辆月成本`。由于所有车辆都会加上相同的常数 1，实际排序主要由“当月预计营收 / 车辆月成本”决定，比例越低评分越低，越优先派遣。

按当前公式，启用车龄权重后，车辆年龄越大，`1 - ((G - F) / 365) × J` 的结果越低，车辆评分越低，系统会更优先派遣车龄更大的车辆。本期车龄权重默认 0%，不影响当前派车排序。

### 2.1 参数口径
1. **已行驶里程 B**：取车辆当前累计行驶里程。
2. **预计退役里程 C**：取车辆资料中配置的预计退役里程；未配置或为 0 时，该车辆不参与自动评分，进入资料待补清单。
3. **当月预计营收 D**：按订单服务日期归属月份统计，包含当月预约中订单金额和当月已完成订单金额。
4. **车辆成本 E**：取车辆资料中配置的每月车辆成本；未配置或为 0 时，该车辆不参与自动评分，进入资料待补清单。
5. **出厂年月 F**：取车辆资料中的出厂年月；缺失时不参与自动评分。
6. **当前年月 G**：按系统当前日期换算。

### 2.2 当前默认计算效果
当前默认里程权重和车龄权重为 0%，营收权重为 100%，因此评分主要由当月预计营收 / 车辆月成本决定。营收达成比例越低，车辆评分越低，越优先派遣。

示例：

```text
已行驶里程 = 2,600
预计退役里程 = 30,000
当月预计营收 = 15,000
车辆成本 = 32,000
里程权重 = 0%
营收权重 = 100%
车龄权重 = 0%

车辆评分 = (2600 / 30000 × 0) + (15000 / 32000 × 1) + (1 - 车辆年龄 × 0)
车辆评分 = 1.47
```

## 3. 订单排序

智能调度先对订单池排序，再逐笔为订单匹配车辆。

| 排序字段 | 排序方式 | 说明 |
| :--- | :--- | :--- |
| 调度占用时长 | 降序 | 同车组内，占用时间越长越优先。 |
| 订单预计金额 | 降序 | 占用时长相同时，金额越高越优先。 |
| 预计取车时间 | 升序 | 占用时长和金额相同时，越临近取车越优先。 |
| 创建时间 | 升序 | 仍相同时，先创建订单优先。 |

调度占用时长按订单预计取车时间、预计还车时间和订单间隔时间计算：

```text
调度占用时长 = (预计还车时间 + 订单间隔时间) - (预计取车时间 - 订单间隔时间)
```

订单排序不再计算独立的订单权重公式。同一车组内，租期越长通常对应更高租金和更高预计里程，直接按调度占用时长优先更符合“大石块优先填空”的排程目标。订单预计金额仅作为占用时长相同时的次级排序条件，避免高价短订单优先切碎低评分车辆的时间轴。

订单预计金额取订单创建时的费用快照，包含租金、增值服务和已匹配行销优惠、优惠券、积分后的订单金额。企业月结订单同样按订单预计金额参与次级排序，不因月结而降低调度资格。

## 4. 车辆池过滤

每处理一笔订单，系统先筛选可用车辆，再计算车辆评分。过滤规则如下：

1. 车辆状态必须可出租。
2. 车辆必须属于订单预订车组。
3. 车辆必须在订单取车门店可交付。本期不做跨门店调拨，其他门店车辆不进入自动派车池。
4. 车辆在订单取还车时间窗内不得存在实际用车中订单、已硬锁订单、维修任务、事故停用任务或其他不可移动任务。
5. 判断时间冲突时，订单前后均叠加 60 分钟订单间隔时间。
6. 车辆证照、保险、年检、保养和安全状态必须满足出租条件。

未通过过滤的车辆不计算评分。

## 5. 贪婪匹配流程

```text
1. 获取参与优化的订单池。
2. 按调度占用时长、订单预计金额、预计取车时间、创建时间排序。
3. 获取当前门店可交付车辆池。
4. 遍历排序后的订单。
5. 对每笔订单筛选可用车辆。
6. 对可用车辆计算车辆评分。
7. 选择评分最低的车辆作为该订单的预分配车辆。
8. 在本轮优化内存中写入该车辆的时间占用。
9. 在本轮优化内存中更新该车辆的预计营收和预计里程。
10. 继续处理下一笔订单。
```

### 5.1 内存模拟更新
同一轮优化中，系统每完成一笔订单分配后，立即更新该车辆的模拟数据：

| 模拟数据 | 更新规则 |
| :--- | :--- |
| 时间占用 | 将订单取车时间、还车时间和前后 60 分钟间隔写入车辆时间轴。 |
| 当月预计营收 | 若订单服务日期归属当月，将订单预计金额计入车辆当月预计营收。 |
| 已行驶里程 | 按订单租期 × 基准日均里程估算，并计入车辆模拟里程。 |

基准日均里程只用于把已分配订单换算为车辆本轮模拟新增里程，进而影响后续车辆评分；不用于计算订单排序权重。订单排序仍按第 3 节的调度占用时长优先规则执行。

内存模拟只影响本轮优化计算。调度员确认应用结果后，系统才将预分配关系写入订单。

## 6. 时间窗判断

车辆可用时间窗按以下口径判断：

```text
车辆占用开始 = 订单取车时间 - 订单间隔时间
车辆占用结束 = 订单还车时间 + 订单间隔时间
```

若任一已硬锁订单、实际用车订单、维修任务、事故停用任务与该时间窗重叠，该车辆不可用于当前订单。

预分配订单在同一轮优化中可以被重排；硬锁订单和实际用车订单不可被自动重排。

## 7. 优化结果

算法输出以下结果：

| 输出项 | 说明 |
| :--- | :--- |
| 订单编号 | 本次参与优化的订单。 |
| 优化前车辆 | 原预分配车辆或未分配。 |
| 优化后车辆 | 本轮算法推荐车辆。 |
| 结果状态 | 使用《智能调度派车规则》中的 `resultStatus` 枚举：已优化、保持原车、未纳入、应用失败。 |
| 变更原因 | 使用《智能调度派车规则》中的 `reasonCode` 和 `reasonText` 枚举，不返回自由文本原因。 |
| 风险状态 | 标记临近锁定、车辆资料缺失、车辆状态变化等风险。 |
| 临近锁定 | 距离硬锁定时间较近的订单需要高亮展示。 |
| 应用结果 | 确认应用时展示已应用、未应用和失败原因。 |

本期手动优化原因只使用以下 8 种：分配待派车、避免车辆冲突、车辆评分更优、保持原车、人工结果保护、临近取车保护、无可用车辆、应用校验失败。

调度页面默认先展示优化结果，不立即写入预分配。调度员确认应用后，系统逐单二次校验订单状态、派车状态、车辆可用性、门店可交付范围、时间冲突、临近取车保护和人工保护，通过校验的订单才更新订单派车信息。

## 8. AI 调度分析与算法关系

AI 调度分析以本文件算法生成的规则优化方案作为系统基线。系统向模型提交调度分析数据包时，需要包含当前规则优化方案、车辆过滤结果、原因码、订单风险标签和不可用车辆原因。

AI 可在以下范围内辅助分析：

1. 对比当前预分配、规则优化方案和 AI 推荐方案的差异。
2. 识别固定公式难以体现的调度风险，例如整备压力、未来连续排程风险、车辆近期异常、晚还风险和门市车源紧张度。
3. 在已通过硬约束过滤的候选车辆中，推荐更适合的车辆。
4. 对无法自动推荐的订单输出人工确认事项。

AI 不得执行以下动作：

1. 将未通过车辆池过滤的车辆放回候选池。
2. 将时间窗冲突车辆推荐给订单。
3. 覆盖已硬锁、人工改派、免费升等或临近取车保护订单。
4. 修改车辆评分公式、订单排序规则或调度规则配置。
5. 直接写入预分配、硬锁或改派结果。

AI 推荐结果应用前，系统仍按本文件第 4 节至第 7 节重新执行候选池过滤、时间窗判断和结果校验。校验失败的 AI 推荐不得应用。

## 9. 失败处理

| 场景 | 处理方式 |
| :--- | :--- |
| 无可用车辆 | 订单不写入预分配车辆，进入人工调度队列。 |
| 车辆资料缺失 | 缺失预计退役里程、车辆成本、出厂年月的车辆不参与自动评分。 |
| 进入临近取车保护窗口 | 进入硬锁定时间点前 2 小时的订单不参与手动批量优化，只能人工改派。 |
| 临近硬锁定仍未分配 | 生成调度告警，调度员人工处理。 |
| 应用结果时车辆状态变化 | 当前车辆跳过，订单重新进入待分配或人工调度队列。 |
| 人工结果保护 | `manual_reassigned` 和 `upgraded` 订单固定不参与自动重排。 |

## 10. 验收标准

| 场景 | 验收标准 |
| :--- | :--- |
| 车辆评分 | 系统按公式计算车辆评分，评分低者优先进入派车候选。 |
| 权重配置 | 默认里程 0%、营收 100%、车龄 0%；三项权重可调整且合计必须为 100%。 |
| 订单排序 | 同车组订单按调度占用时长从长到短优先处理；占用时长相同时，再按预计金额、预计取车时间和创建时间排序。 |
| 基准日均里程 | 基准日均里程只用于分配后更新车辆模拟里程，不用于订单排序权重计算。 |
| 门店限制 | 其他门店车辆不进入自动派车池，不生成跨门店调拨任务。 |
| 时间冲突 | 车辆时间窗叠加前后 60 分钟间隔后无冲突才允许匹配。 |
| 动态更新 | 同一轮优化中，已分配车辆的模拟营收、模拟里程和时间占用即时更新。 |
| 预览应用 | 未确认应用优化结果前，不改变订单派车信息。 |
| 无车可派 | 订单进入人工调度队列，并在调度页面展示风险。 |
| AI 基线对比 | AI 调度分析必须以系统规则优化方案作为基线输入，输出与基线方案的差异和原因。 |
| AI 推荐校验 | AI 推荐车辆必须重新经过车辆池过滤、时间窗判断和应用前二次校验；不通过校验时不得应用。 |
