## 1. 文档概述

### 1.1 背景
门市租车采用车组预订模式。客户下单时锁定车组库存，不立即锁定具体车牌；系统需要在取车前根据订单价值、车辆可用性和车辆经营数据完成车辆预分配与硬锁定，减少门店人工派车成本，并让车辆使用强度保持均衡。

### 1.2 目标
1. **提升派车效率**：订单创建后先锁定车组库存，系统在调度窗口内自动生成车辆预分配方案。
2. **保障取车交付**：取车前按配置时间硬锁具体车辆，避免临近取车时车辆被其他订单占用。
3. **均衡车辆经营数据**：按车辆评分低者优先派遣，让营收、里程、车龄等指标通过可配置权重参与派车。
4. **保留人工干预**：调度员可预览优化结果、保护人工改派订单，并在异常场景下手动改派。
5. **支持 AI 辅助分析**：已配置 AI 模型的租户可在调度页面发起 AI 调度分析，系统向模型提交调度分析数据包和平台内置系统提示词，由 AI 输出风险诊断和推荐动作；推荐结果必须经过系统二次校验和人工确认后才可应用。

## 2. 核心概念

| 概念 | 说明 |
| :--- | :--- |
| 车组库存锁定 | 新建订单成功后锁定对应车组在取还车时间段内的可售库存，不绑定具体车牌。 |
| 预分配 | 系统为订单预选具体车辆，订单派车状态为 `pre_assigned`；预分配仍可参与后续智能优化。 |
| 硬锁定 | 取车前达到锁定时间后，系统对该订单执行一次自动分派或预分配有效性复核，通过后将具体车辆绑定为硬锁定，订单派车状态为 `locked`。 |
| 人工改派 | 调度员、店长或管理员手动调整车辆，订单派车状态为 `manual_reassigned`。 |
| 免费升等 | 因门店责任、车辆不可交付或补偿场景，将订单派到更高车组且不向客户收取差价，订单派车状态为 `upgraded`。 |
| 人工调度队列 | 系统无法自动派车、车辆临时不可用或临近取车仍未锁车时，订单进入人工处理范围；订单主状态不因此变化。 |
| AI 调度分析 | 租户完成 AI 模型配置后，系统基于调度数据包和系统提示词调用模型，生成风险诊断、推荐动作和解释说明；AI 不直接修改订单、车辆或排程。 |

### 2.1 状态轴线定义

系统判断车辆是否可以参与门市租赁调度时，不得只读取一个“车辆状态”字段，必须同时判断以下四条独立状态轴线：

| 状态轴线 | 典型取值 | 用途 |
| :--- | :--- | :--- |
| 车辆启用状态 | `enabled`、`disabled` | 控制车辆是否允许参与新订单销售、自动派车、人工改派、硬锁定和确认发车。 |
| 车辆营运状态 | `idle`、`renting`、`controlled` | 控制车辆当前时点是否空闲、已在用车中或处于内部管控中。 |
| 订单派车状态 | `pending`、`pre_assigned`、`locked`、`manual_reassigned`、`upgraded`、`dispatch_manual_required` | 控制该订单与具体车辆的绑定关系和是否允许继续优化。 |
| 订单主状态 | `pending_payment`、`reserved`、`pickup_overdue`、`inspecting`、`renting`、`renewing` 等 | 控制订单当前业务阶段，以及是否允许派车、取车、还车或进入结算。 |

其中，车辆启用状态和车辆营运状态属于车辆档案维度；订单派车状态和订单主状态属于订单维度。四者必须并行判断，不能互相替代。

### 2.2 车组库存与具体车辆占用关系

门市租赁库存按“订单占用车组库存、派车状态绑定具体车辆”拆开处理。

1. 每笔有效门市租赁订单固定占用 1 个车组库存名额。
2. `pre_assigned`、`locked`、`manual_reassigned`、`upgraded` 只表示订单已经绑定具体车辆，不在车组库存层额外重复扣减。
3. 同车组改派、同车组换车只释放原车辆并占用新车辆，不改变订单原有车组库存占用。
4. 改车组、续租、改租期会改变车组或时间窗，必须重新校验库存，并在同一事务内更新占用。
5. 还车验车提交后，订单不再作为车组库存占用，但车辆在还车确认完成并恢复可售前，仍通过车辆 `controlled` 或还车待整备任务减少可服务车辆数。

## 3. 默认配置

| 配置项 | 默认值 | 说明 |
| :--- | :--- | :--- |
| 车辆排程看板锁定时间 | 默认取车前 2 小时，可配置 | 达到时间后，系统对每笔订单执行一次自动分派或预分配有效性复核，通过后转为硬锁定。 |
| 订单间隔时间 | 60 分钟 | 同一车辆前后两单之间必须预留的清洁、检查和整备时间。 |
| 基准日均里程 | 150 km/天 | 用于派车模拟时估算订单给车辆带来的新增里程，不用于同车组订单排序权重。 |
| 里程权重 | 0% | 当前默认不以里程作为主要派车因素，后台可调整。 |
| 营收权重 | 100% | 当前默认优先均衡车辆当月预计营收，后台可调整。 |
| 车龄权重 | 0% | 当前默认不以车龄作为主要派车因素，后台可调整。 |
| 周期优化时间 | 每日 02:00 | 系统自动对未来默认优化窗口内的订单进行预分配重排。 |
| 默认优化窗口 | 未来 7 天 | 调度页面支持手动选择未来 3 天、7 天、14 天。 |

里程权重、营收权重、车龄权重三项合计必须等于 100%。本期配置为门市租车全局配置，不按门店、车组或企业客户单独配置。

### 3.1 订单排序口径

智能调度在同一车组内不再计算独立的订单权重公式。订单池先按调度占用时长从长到短排序，再按预计金额、预计取车时间和创建时间排序。

```text
调度占用时长 = (预计还车时间 + 订单间隔时间) - (预计取车时间 - 订单间隔时间)
```

处理口径：

1. 同车组内，租期越长通常对应更高租金和更高预计里程，直接按调度占用时长排序即可表达“大石块优先填空”。
2. 预计金额只作为占用时长相同时的次级排序条件，不作为第一排序条件。
3. 基准日均里程只用于订单分配后更新车辆本轮模拟新增里程，进而影响下一笔订单选择车辆时的车辆评分。
4. 该口径用于周期优化、手动优化和 AI 调度分析的系统基线方案。

## 4. 派车状态流转

| 场景 | 触发动作 | 派车状态 | 车辆绑定结果 |
| :--- | :--- | :--- | :--- |
| 新建订单成功 | 客户自付订单创建成功，或企业月结订单创建成功 | `pending` | 锁定车组库存，不绑定车牌；`pending_payment` 仅保留车组库存，不进入车辆预分配。 |
| 周期优化完成 | 系统每日自动优化，生成预分配方案 | `pre_assigned` | 写入预分配车辆，可被后续优化调整。 |
| 手动优化应用 | 调度员在排程看板确认应用优化结果 | `pre_assigned` | 写入预分配车辆，可按策略继续参与优化。 |
| 达到硬锁定时间 | 当前时间进入取车前锁定窗口 | `locked` 或 `dispatch_manual_required` | 系统先自动分派或校验预分配车辆；成功时绑定具体车牌，不再参与自动优化；失败时进入人工调度队列。 |
| 即时单创建 | 取车时间已小于锁定窗口 | `locked` | 系统直接在可用车辆池中匹配并锁定车辆。 |
| 人工改派 | 调度员或门店人员手动调整车辆 | `manual_reassigned` | 释放原车辆，锁定新车辆。 |
| 免费升等 | 门店责任或补偿场景确认升等 | `upgraded` | 锁定升等车辆，写入免费升等原因。 |
| 取消、爽约、关闭订单 | 订单进入终态 | 原派车状态保留记录 | 释放车组库存、具体车辆绑定和派车关系。 |
| 取车确认发车 | 取车流程完成 | `locked` | 车辆进入实际用车占用，不再参与优化。 |

## 5. 调度执行时机

### 5.1 下单响应期
1. 新建订单成功后，系统只校验并锁定车组库存。
2. 系统不在下单瞬间锁定具体车牌。
3. 订单派车状态写入 `pending`。
4. `pending_payment` 订单在待支付资源保留时长内只保留车组库存，不进入周期优化、手动优化、预分配和硬锁定；补收预授权成功进入 `reserved` 后再参与派车。
5. 待支付资源保留超时后，订单进入 `cancelled`，系统释放车组库存和派车关系。

### 5.2 周期优化期
1. 系统每日 02:00 自动执行周期优化。
2. 优化范围默认未来 7 天，可通过调度页面手动选择未来 3 天、7 天、14 天。
3. 周期优化只处理未硬锁定、未实际发车、未进入终态的订单。
4. 周期优化完成后，系统生成预分配结果，订单派车状态更新为 `pre_assigned`。

### 5.3 手动优化期
调度页面支持调度员手动发起智能优化。手动优化只处理未硬锁定、未取车、未进入异常/结算/终态的预分配订单。手动优化不处理取车中、用车中、已硬锁、人工改派、免费升等、逾期未取等订单。

企业月结只影响订单支付和结算方式，不影响车辆调度资格。企业月结订单只要主状态和派车状态符合手动优化范围，即可参与手动优化。

手动优化只保留一个可选控制项：

| 控制项 | 规则 |
| :--- | :--- |
| 优化时间窗口 | 支持未来 3 天、7 天、14 天。 |

手动优化固定执行以下系统规则：

| 固定规则 | 处理口径 |
| :--- | :--- |
| 默认先预览 | 点击“开始优化”后只生成优化结果，不立即写入订单。调度员点击“确认应用优化结果”后才更新预分配车辆。 |
| 订单排序 | 同车组订单先按调度占用时长降序处理；占用时长相同时，再按预计金额、预计取车时间和创建时间排序。 |
| 固定保护人工结果 | `manual_reassigned` 和 `upgraded` 订单不参与手动批量优化，只能通过订单改派流程单独处理。 |
| 减少车辆变动 | 当推荐车辆相对原车辆的评分改善不明显，且原车辆仍可交付、无冲突、无风控风险时，系统保留原预分配车辆。 |
| 临近取车保护 | 进入硬锁定时间点前 2 小时的订单不参与手动批量优化，只能人工改派。硬锁定时间点按门市租车全局规则中的车辆排程看板锁定时间计算。 |

#### 5.3.1 手动优化结果状态和原因

手动优化结果中的“变更原因”必须使用固定枚举，不允许返回自由文本作为业务原因。接口返回 `resultStatus`、`reasonCode`、`reasonText`；页面展示 `reasonText`。如需展示补充说明，可返回 `reasonDetail`，但 `reasonDetail` 不参与业务判断。

| 结果状态 `resultStatus` | 中文显示 | 说明 |
| :--- | :--- | :--- |
| `optimized` | 已优化 | 本次优化调整了预分配车辆，或为未分配订单补齐车辆。 |
| `unchanged` | 保持原车 | 本次优化后仍保留原预分配车辆。 |
| `skipped` | 未纳入 | 订单因保护规则或无可用车辆未参与本次批量优化。 |
| `failed` | 应用失败 | 确认应用时二次校验未通过，结果未写入订单。 |

| 原因编码 `reasonCode` | 页面显示 `reasonText` | 适用状态 | 使用场景 |
| :--- | :--- | :--- | :--- |
| `ASSIGN_PENDING` | 分配待派车 | `optimized` | 订单原派车状态为 `pending`，本次找到可用车辆并写入预分配。 |
| `REPLACE_CONFLICT` | 避免车辆冲突 | `optimized` | 原预分配车辆存在时间冲突、交付风险、后续硬锁冲突或不可移动任务影响。 |
| `REPLACE_SCORE` | 车辆评分更优 | `optimized` | 推荐车辆按当前评分规则更适合派遣，且调整收益达到系统阈值。 |
| `KEEP_CURRENT` | 保持原车 | `unchanged` | 原预分配车辆仍可交付，且调整收益不足以触发换车。 |
| `SKIP_MANUAL` | 人工结果保护 | `skipped` | 订单派车状态为 `manual_reassigned` 或 `upgraded`。 |
| `SKIP_NEAR_LOCK` | 临近取车保护 | `skipped` | 订单已进入硬锁定时间点前 2 小时保护窗口。 |
| `SKIP_NO_VEHICLE` | 无可用车辆 | `skipped` | 同门店、同车组、同时间窗内没有可用车辆。 |
| `APPLY_VALIDATION_FAILED` | 应用校验失败 | `failed` | 确认应用时订单状态、派车状态、车辆状态、时间冲突或人工保护条件已发生变化。 |
| `AI_RISK_AVOIDANCE` | AI 风险规避 | `optimized` | AI 基于整备压力、未来冲突、车辆异常或晚还风险，推荐不同于基线方案的车辆；应用前仍需系统校验。 |
| `AI_MANUAL_REVIEW` | AI 建议人工确认 | `skipped` | AI 无法在硬约束内给出明确车辆，或发现需要人工判断的异常。 |
| `AI_DATA_INSUFFICIENT` | AI 数据不足 | `skipped` | 调度分析数据包缺少关键字段，AI 无法完成可靠判断。 |

手动优化确认应用时，系统逐单进行二次校验。通过校验的订单写入新的 `pre_assigned` 车辆；未通过校验的订单跳过并展示失败原因，不影响其他订单应用。

| 二次校验项 | 通过条件 |
| :--- | :--- |
| 订单状态 | 订单仍为 `reserved`。 |
| 派车状态 | 派车状态仍为 `pending` 或 `pre_assigned`。 |
| 临近取车保护 | 未进入硬锁定时间点前 2 小时保护窗口。 |
| 车辆状态 | 推荐车辆仍为可出租、可交付状态。 |
| 门店范围 | 推荐车辆仍在订单取车门店所属车行范围内、且在订单取车门店可交付。本期不跨门店调拨，不跨车行派车。 |
| 时间冲突 | 推荐车辆在订单取还时间窗及订单间隔时间内无冲突。 |
| 人工保护 | 订单未被人工改派、免费升等或硬锁。 |

### 5.4 硬锁定期
1. 当前时间达到规则配置的车辆排程看板锁定时间时，系统必须针对该订单执行一次自动分派任务。
2. 订单已有预分配车辆时，系统先复核该车辆是否仍满足车组、门店、时间窗、订单间隔、车辆状态、证照保险、能源清洁等硬锁条件。
3. 预分配车辆复核通过时，系统将预分配车辆转为硬锁定车辆，订单派车状态为 `locked`。
4. 预分配车辆复核失败或订单没有预分配车辆时，系统立即在订单取车门店可交付车辆池中重新匹配可用车辆；匹配成功后直接硬锁。
5. 自动分派失败时，订单主状态不变化，派车状态标记为 `dispatch_manual_required`，进入人工调度队列并在调度看板展示告警。
6. 硬锁定后订单不再参与周期优化和手动优化，只能通过订单改派流程调整。
7. 取车时必须存在已硬锁车辆；未锁定车辆时不允许确认发车。

### 5.5 即时单补救期
1. 订单创建时已进入车辆排程看板锁定时间窗口，系统跳过预分配，直接执行即时匹配。
2. 即时匹配成功后，订单派车状态直接写入 `locked`。
3. 即时匹配失败时，订单进入人工调度队列，并在调度看板显示调度告警。

### 5.6 AI 调度分析期

AI 调度分析不改变订单派车状态。调度员在车辆排程看板发起 AI 分析后，系统按当前筛选范围和 3/7/14 天分析窗口生成调度分析数据包，并携带平台内置系统提示词调用租户配置的模型提供商。

AI 调度分析固定执行以下规则：

| 规则 | 处理口径 |
| :--- | :--- |
| 租户配置前置 | 租户必须已开通 AI 调度分析能力，并在营运平台完成模型提供商、API Key 和模型名称配置。 |
| 系统提示词前置 | 每次调用模型时，系统必须提交平台内置系统提示词，明确 AI 是门市租赁调度分析 Agent，并声明同车行、车组、车辆状态、时间窗、硬锁、人工保护等硬约束不得突破。 |
| 系统候选池前置 | 调用模型前由系统先完成硬约束过滤，AI 推荐车辆必须来自该订单合法候选车辆列表；不在候选池内的推荐不得应用。 |
| 数据最小化 | 系统只提交调度分析所需的订单、车辆、门市、据点、排程、规则、基线方案和预计算风险，不提交手机号、证件号、支付资料、API Key 或与调度无关的敏感信息。 |
| 只输出推荐 | AI 只能输出风险诊断、推荐动作、推荐原因、信心等级和需要人工确认的事项，不得直接执行派车、硬锁、改派、取消或支付相关动作。 |
| 系统二次校验 | AI 推荐结果必须通过 JSON Schema 校验、车辆硬约束校验、订单状态校验、时间窗冲突校验、人工保护校验和权限校验，未通过校验的推荐不得应用。 |
| 人工确认应用 | 调度员确认后，系统按现有预分配应用逻辑逐单写入；应用失败不影响其他订单。 |
| 规则优化兜底 | AI 未配置、调用失败、返回格式错误或超时时，现有规则智能优化仍可继续使用。 |

AI 调度分析详细规则见《AI 智能调度分析 Agent》。

## 6. 参与优化的订单范围

| 订单条件 | 是否参与自动优化 | 处理口径 |
| :--- | :--- | :--- |
| `mainStatus=reserved` 且 `dispatchStatus=pending` | 是 | 作为待分配订单进入优化池。 |
| `mainStatus=reserved` 且 `dispatchStatus=pre_assigned` | 是 | 可被周期优化或手动优化重新编排。 |
| `mainStatus=pending_payment` | 否 | 客户自付订单资金未确认，只在待支付资源保留时长内占用车组库存，不锁定具体车辆。 |
| `dispatchStatus=locked` | 否 | 已硬锁具体车辆，只能通过订单改派流程调整；车辆事故、维修、停运等不可交付异常发生且订单尚未进入取车中或用车中时，可由异常改派任务解除占用并重新分派。 |
| `dispatchStatus=manual_reassigned` | 否 | 人工改派结果固定保护，不参与周期优化和手动批量优化。 |
| `dispatchStatus=upgraded` | 否 | 免费升等结果固定保护，不参与周期优化和手动批量优化。 |
| 已进入硬锁定时间点前 2 小时保护窗口 | 否 | 临近取车订单不参与手动批量优化，只能人工改派。 |
| `mainStatus=pickup_overdue` | 否 | 已超过取车时间，保持原车辆绑定或进入人工调度队列。 |
| `mainStatus=inspecting/renting/renewing/overdue/accident_processing` | 否 | 已进入取车、用车或异常履约阶段，不参与预分配优化。 |
| `mainStatus=settlement_pending/payment_due/refund_pending/completed/cancelled/no_show/rejected/closed` | 否 | 结算中或终态订单不参与调度优化。 |

## 7. 车辆可用性过滤

系统为订单筛选车辆时，必须先通过硬约束过滤。未通过硬约束的车辆不得进入评分排序。门市租赁候选池判断必须同时读取车辆启用状态、车辆营运状态、订单时间窗占用、派车状态和订单主状态。

### 7.1 状态轴线过滤

| 判断维度 | 规则 |
| :--- | :--- |
| 车辆启用状态 | 车辆必须为 `enabled` 才能参与新订单销售、自动派车、人工改派、硬锁定和确认发车。`disabled` 车辆仅允许继续完成已在履约中的还车。 |
| 车辆营运状态 | `idle` 允许作为当前即时可交付车辆；`renting` 不允许参与重叠时间窗派车，但在前序订单结束且缓冲满足后可参与未来无冲突订单候选池；`controlled` 在管控解除前不得参与预分配、硬锁定和确认发车。 |
| 订单派车状态 | 仅 `pending`、`pre_assigned` 订单参与周期优化和手动优化；`locked` 只做硬锁复核；`manual_reassigned`、`upgraded` 默认受人工保护。 |
| 订单主状态 | `pending_payment` 不进入派车；`reserved`、`pickup_overdue` 可进入派车；`inspecting`、`renting`、`renewing`、`overdue`、`settlement_pending` 及终态不参与预分配优化。 |

### 7.2 候选池业务过滤

| 过滤项 | 规则 |
| :--- | :--- |
| 车组匹配 | 车辆必须属于订单预订车组。同组车辆优先；免费升等不进入常规自动匹配，由人工确认。 |
| 门店匹配 | 车辆资产所属车行必须与订单取车门店所属车行一致，且车辆必须在订单取车门店可交付。本期不做跨门店调拨，不自动生成调拨任务，不把其他门店或其他车行车辆纳入自动派车池。 |
| 时间窗冲突 | 车辆在订单取车时间至还车时间之间不得存在用车中订单、已硬锁订单、被管控任务或其他不可移动任务。 |
| 整备缓冲 | 判断车辆可用时，订单前后均需叠加 60 分钟订单间隔时间。 |
| 证照与保险 | 车辆证照、保险、年检等有效性必须满足可出租要求。 |
| 保养与安全 | 车辆已达到保养、检修或风控停用条件时，不得参与自动派车。 |
| 能源与清洁 | 车辆油量/电量、清洁状态未达到取车交付标准时，不得进入硬锁定；可在预分配阶段展示风险标签。 |

### 7.3 候选池与动作矩阵

| 车辆状态组合 | 预分配 | 硬锁复核 | 取车发车 | 还车完成 | 说明 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `enabled + idle` | 允许 | 允许 | 允许 | 允许 | 普通可履约车辆。 |
| `enabled + renting` | 仅允许参与未来无冲突订单预分配 | 仅当前序占用结束且进入锁定窗口时允许 | 不允许 | 允许 | 必须同时检查前序订单结束时间和订单间隔时间。 |
| `enabled + controlled` | 不允许 | 不允许 | 不允许 | 允许 | 管控未解除前不恢复履约资格。 |
| `disabled + idle` | 不允许 | 不允许 | 不允许 | 允许 | 禁用阻断新销售、派车和发车。 |
| `disabled + renting` | 不允许 | 不允许 | 不允许 | 允许 | 已在履约中的订单继续完成还车。 |
| `disabled + controlled` | 不允许 | 不允许 | 不允许 | 允许 | 维持不可履约状态，等待后续人工处理。 |

## 8. 异常处理

### 8.1 到达硬锁定时间无车可派
1. 系统保留订单主状态，不自动取消订单。
2. 订单进入人工调度队列。
3. 调度看板展示“无车可派”告警。
4. 本期不自动发起跨门店调拨。
5. 本期不自动免费升等；调度员可人工选择免费升等，并写入原因和操作记录。

### 8.2 已分配车辆临时不可用
1. 若车辆不可用来源为禁用、事故、维修、停运、待定损、被管控中或证照保险不满足交付条件，系统需要扫描该车后续仍未开始履约的订单。
2. 对 `pending`、`pre_assigned`、`locked` 且主状态仍为 `reserved` 或 `pickup_overdue` 的订单，系统立即执行二次校验；`pending`、`pre_assigned` 订单释放原预分配结果并尝试自动改派，`locked` 订单在未进入 `inspecting` 前解除硬锁并尝试自动改派。自动改派成功后写入新车辆和操作记录。
3. 对 `manual_reassigned`、`upgraded` 的订单，系统保留人工结果保护，生成调度告警并进入人工调度队列，除非调度员在订单改派流程中主动调整。
4. 对已经进入 `inspecting`、`renting`、`renewing`、`overdue`、`accident_processing` 的订单，系统不自动改派，必须由当前履约流程或订单改派流程处理。
5. 自动改派失败时，订单主状态不变化，派车状态或风险标记增加 `dispatch_manual_required`，调度看板展示“车辆不可交付，需人工改派”。

### 8.3 延长用车导致后续订单冲突
1. 当前订单申请延长用车时，系统先检查原车辆后续硬锁订单。
2. 若后续订单可改派到同门店、同车组、无冲突车辆，则允许延长，并改派后续订单。
3. 若后续订单无法改派，则拒绝自助延长；门店人员可人工协调并按《订单改派》规则处理。

### 8.4 改还车门市导致原还车门市后续冲突
1. 当前订单申请改还车门市时，系统先检查原还车门市是否存在占用当前车辆的后续预约、待取车或硬锁订单。
2. 若后续订单可改派到原还车门市下同车组或可升级车辆，则允许先快捷改派后续订单，再确认当前订单转单。
3. 若后续订单无法改派，则当前订单不能直接转单；门店需先人工协调后续订单占用，或取消本次改还车门市。

### 8.5 取车时未锁定车辆
1. 取车流程拦截确认发车。
2. 门店需先完成派车或人工改派。
3. 完成硬锁定后，取车流程继续。

### 8.6 还车后的车辆状态回写
1. 还车验车提交后，车辆结束用车中占用并进入内部处理阶段；订单在 `settlement_pending`、`payment_due`、`refund_pending` 期间，车辆不得重新进入新的派车候选池。
2. 订单确认还车后，如车辆仍为 `enabled` 且无需清洁、保养、调度或其他内部处理，车辆营运状态回写为 `idle`。
3. 订单确认还车后，如车辆仍需清洁、保养、调度、待整备或其他内部处理，车辆营运状态保持 `controlled`，直到管控解除。
4. 车辆在还车完成时若已 `disabled`，则继续保持禁用，不恢复新订单派车资格。

## 10. 验收标准

| 场景 | 验收标准 |
| :--- | :--- |
| 新建订单 | 创建成功后锁定车组库存，派车状态为 `pending`，不显示具体车牌；待支付订单在资源保留时长内只保留车组库存，不参与车辆预分配。 |
| 周期优化 | 系统对符合条件的 `reserved` 订单生成预分配车辆，派车状态更新为 `pre_assigned`。 |
| 手动优化 | 调度员可选择 3/7/14 天窗口并预览优化结果，确认后写入预分配。 |
| 订单排序 | 同车组订单按调度占用时长降序处理；基准日均里程只用于更新车辆模拟里程，不用于订单排序权重。 |
| 人工保护 | 人工改派和免费升等订单固定不参与周期优化和手动批量优化。 |
| 临近取车保护 | 进入硬锁定时间点前 2 小时的订单不参与手动批量优化。 |
| 应用校验 | 确认应用优化结果时逐单二次校验；通过的订单写入预分配，未通过的订单跳过并展示失败原因。 |
| 硬锁定 | 到达规则配置的车辆排程看板锁定时间时，预分配车辆转为硬锁定，派车状态更新为 `locked`。 |
| 即时单 | 订单创建时已进入车辆排程看板锁定时间窗口，订单直接即时匹配并硬锁车辆。 |
| 跨门店车辆 | 其他门店车辆和其他车行车辆不进入自动派车池，不生成跨门店调拨任务。 |
| 无车可派 | 订单进入人工调度队列，调度看板展示告警，不自动取消订单。 |
| 取车拦截 | 未硬锁车辆时，不允许确认发车。 |
| 禁用车辆 | `disabled` 车辆不参与预分配、硬锁和发车，但已在履约中的订单允许完成还车。 |
| 租用中未来排程 | `renting` 车辆不参与重叠时间窗派车，但在前序订单和缓冲结束后，可参与未来无冲突订单的预分配。 |
| 还车回写 | 还车完成后，车辆按 `idle / controlled / disabled` 规则回写状态，不允许在结算未完成时重新进入候选池。 |
| AI 模型配置 | 租户未开通或未配置模型时，车辆排程看板 AI 分析入口不可应用；租户已配置模型后，系统可按《AI 智能调度分析 Agent》生成调度分析数据包并调用模型。 |
| AI 推荐校验 | AI 返回结果通过 JSON Schema 和业务硬约束校验后才允许展示为可应用推荐；未通过校验的推荐只展示失败原因。 |
| AI 应用结果 | 调度员确认应用 AI 推荐时，系统仍按现有应用优化结果逻辑逐单二次校验并写入预分配，不允许 AI 直接修改订单、车辆或排程。 |
